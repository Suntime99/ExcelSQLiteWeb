# 下游数据契约（v1）

目标：让“查询/统计/透视/图表/导出”等下游模块**只依赖业务库（SQLite）内的表/视图**，与源文件（Excel/CSV）解耦，保证可复现、可审计、可版本化。

---

## 1. 数据分层约定（强烈推荐）

### 1.1 raw 层（保真快照）
- 表命名：`raw__{主表名}`（例如：`raw__Sales|明细`）
- 语义：首次清洗/重建时的“原始快照”，用于：
  - 回溯对比（验证清洗前后差异）
  - 出问题时快速恢复（重新生成 clean / vw_）
  - 归档压缩（可移出主库）
- 下游模块：**禁止直接面向 raw__**（除非做对账/审计工具）

### 1.2 clean 层（清洗后的稳定事实表）
- 表命名：`clean__{主表名}`（例如：`clean__Sales|明细`）
- 语义：清洗输出默认落到 clean__，避免覆盖与叠加污染
- 下游模块：允许面向 `clean__*` 做分析（推荐）

### 1.3 view 层（业务口径层）
- 视图命名：强制 `vw_` 前缀（例如：`vw_rel_sales`、`vw_fact_sales`）
- 语义：用 SQL 固化口径（多表关联、字段衍生、过滤口径等）
- 下游模块：**优先面向 vw_**（最推荐）

---

## 2. 下游模块只读原则

下游模块（单表查询、多表查询、统计、透视、图表、搜索等）应遵循：
- **只读**：仅执行 `SELECT/WITH...SELECT`
- **只认业务库表/视图**：不直接读取源文件
- **只认稳定对象**：优先 `vw_` 与 `clean__*`，避免直接引用中间临时表

写入类操作（导入、清洗、合并、Upsert、归档、修订点）统一归口在：
- 方案管理（数据接入/导入任务）
- 数据清洗（执行清洗/归档 raw）
- 建模管理（派生视图管理）

---

## 3. 表名与显示名（Alias）

为避免“改物理表名导致关联/视图/模板失效”，约定：
- 物理表名（SQLite name）：导入后尽量稳定，不随 UI 随意重命名
- 显示名/注释名（Alias）：可随时修改，用于 UI 下拉展示，格式：`显示名（物理名）`

---

## 4. 版本链与修订点（建议口径）
- 导入、清洗等关键动作前可自动创建修订点（备份库）
- 版本号建议与导入批次/修订点绑定（例如：`rev-YYYYMMDDHHmmss`）
- 视图版本号：每次更新 SQL 自增（已实现）

---

## 5. 基础库（Base DB）挂载约定
- 基础库以 `ATTACH DATABASE ... AS base` 的形式挂载
- 下游引用基础库对象时使用：`base.TableName` / `base.vw_xxx`
- 业务库（main）与基础库（base）应尽量避免同名对象
