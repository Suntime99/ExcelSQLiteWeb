## 1) 职责边界（必须遵守的硬规则）

### 1.1 核心对象（统一名词）
- **Active**：当前活动项目、当前活动 DB 连接、当前主表、当前模式
- **Config**：项目配置（源文件关系/导入策略/建模配置/偏好）
- **DB**：项目数据库文件（SQLite 产物）
- **Objects**：数据库对象（表/视图/索引…）
- **Model**：语义模型（关系识别结论、MTJ 草稿、锁定 SQL、派生视图定义）

### 1.2 页面 = Owner（唯一主责）
- **项目中心（Project Hub）**：Owner=**Active**（切换/加载/进入工作状态）
- **项目信息（Project Info）**：Owner=**只读摘要**（展示 Active/Config/DB/Model/Objects 的状态）
- **项目编辑（源文件关系）（Project Editor）**：Owner=**Config**（尤其 Source Bindings）
- **建模管理（Model Mgmt）**：Owner=**Model**（写 Config 的模型子树；可选生成 View 对象）
- **数据库管理（DB Mgmt）**：Owner=**DB + Active(DB)**（导入/重导/加载/备份）
- **对象/索引管理（Object/Index Mgmt）**：Owner=**Objects**（删建改名/依赖/索引）

> 判定法：  
> 改配置→项目编辑/建模；生成/替换 SQLite 文件→数据库管理；切换当前上下文→项目中心；改表/视图/索引→对象/索引管理。

## 3) “先调”落地清单（尽量不大改架构）

> 目标：不做“页面大整合重写”，但把用户心智最乱的地方先调顺：  
> - **同义按钮去重**  
> - **配置/数据库/加载 三类动作分组**  
> - **文案按状态机变化**  
> - **复杂策略向导化（最小两步）**

### 3.1 文案与动作语义收敛（立刻做，收益最大）

#### A) “加载到SQLite数据库”改名（避免误解）
现按钮：`🗄️ 加载到SQLite数据库`
建议改为（更准确）：
- **“导入到项目数据库”**（推荐）
- 若未绑定项目（临时库）：**“导入到临时数据库”**

理由：用户理解“加载”是打开 DB，不是把 Excel 写进 DB。

#### B) “重建数据库”按状态改名（你提出的反人类点）
现按钮：`♻ 重建数据库`
建议：
- **首次**（无 Project DB）：按钮显示 **“保存数据库”** 或 “生成数据库”
- **非首次**（已有 Project DB）：按钮显示 **“重导数据库”**（或“重建/重导数据库”）

并在按钮 hover/提示中明确：
> 重导会覆盖项目数据库；若存在未提交变更将丢失（可选先备份）。

#### C) “保存为项目”与“自动保存”位置调整
把 `项目名称 + 自动保存 + 保存为项目` 作为 **项目配置区（Config）** 的固定头部，不要跟 DB 操作按钮混排。

并满足你提出的“项目名默认”：
- 新建时自动给默认名（用户想改再改）

---

### 3.2 `file-wizard` 内部重排（不拆页面，但先分出 3 块）

建议把 `fwPage_import` 的 UI 结构改成三段（视觉上就是三张卡/三行分组）：

#### ① 项目配置（Config）
- 项目名称（默认生成）
- 自动保存（开关）
- 保存项目（保存配置/关系）
- **“源文件清单”入口/提示**（用户知道先改关系再导入）

#### ② 数据库操作（DB lifecycle）
- 导入到项目数据库 / 导入到临时数据库（根据是否绑定项目决定）
- 保存数据库 / 重导数据库（按状态变文案）
- 高级开关：导入前清空、导入前修订点（备份）

#### ③ 导入策略（Wizard 化的最小方案）
> 你现在的 UI 同时存在：`写入方式(覆盖/追加)`、`追加策略(append/dedupe/upsert)`、`合并生成表/生成关联视图`，这对普通用户等同于“没有设计”。  
> 先调阶段不必大改后端，先把**入口做成向导**，并在 UI 上“只让用户做一个选择”，其它选项跟随显示/隐藏即可。

**Step 1：选择导入策略（单选）**
- `重建（全量重导）`：等价于“清空后覆盖导入”
- `追加（Append）`：等价于“保留原表，新增行”
- `合并（Merge/Upsert）`：需要键字段 + 冲突策略（至少 1 个键）

**Step 2：策略参数（按 Step1 显示）**
- 重建：只保留“清空/修订点/导入模式（text/smart）”
- 追加：追加策略（append/dedupe/upsert）+ 键字段（去重/upsert 必填）
- 合并/关联：输出表名 / 视图名（统一要求 vw_ 前缀）+ “生成”按钮

> 先调落地要点：把现在的 `fwIngestTab_base/append/merge` 变成“向导 Step2 的不同面板”，而不是“用户随便点哪个 tab 都能同时生效”。

### 3.3 “选定主表”归位（Active 默认值，而非导入选项）
现位置：在导入区底部（`sqliteMainTableSelect`）。

建议：
- 文案改为：**“设置当前主表（影响默认分析/建模入口）”**
- 按钮/链接：`去建模管理`、`去多表关联`（把它变成主控入口，而不是导入参数）
- 规则：切主表只改 **Active**，不触发导入、不改 Config（除非用户点“保存主表到项目配置”）

---

### 3.4 项目中心 `project-hub` 的“先调”建议（不动结构，补状态与入口）

你现在的三视图（overview/active/config）方向是对的，先调建议是“补齐状态提示 + 收敛入口”：

1) **把“数据接入”改名为“项目编辑/数据接入”**（避免用户以为只是导入）
2) `清理临时数据库` 建议挪到 **数据库管理 → 高级/维护**，项目中心保留一个“快捷入口”即可（不要放顶栏主按钮）
3) 在“当前活动项目”卡片里增加 3 个状态灯（文字即可）：
   - 配置：已保存/有未保存更改
   - 数据库：未生成/已生成（最后导入时间）
   - 运行时：未加载/已加载（当前连接）

> 这样用户一眼就知道“我改的是配置还是库？现在有没有加载？”

---

### 3.5 建模管理 `model-management` 的“先调”建议
你当前定位“派生视图（vw_）是核心产出”非常对，先调只建议两点：
1) “派生视图”文案统一改成：**业务视图（vw_）**（降低技术感）
2) 把右侧编辑器的按钮语义进一步明确：
   - `保存视图` = 写入项目配置（Model/Config）
   - `发布到数据库`（可新增按钮）= 生成/更新 DB View（Objects）——危险操作需确认，并写审计

> 先调阶段可以先不做“发布”按钮，只把概念与落盘位置写清楚；但 UI 要避免让用户误以为“保存视图=数据库已更新”。

---

### 3.6 数据库对象管理 `db-object-manager` 的“先调”建议
该页属于 **Objects 运维**，先调主要是“风险收纳与引用更新提示”：
1) 把 `删除选中` 标成危险操作，并在确认弹窗里显示：
   - 选中对象数量、包含视图/表数量
   - 若能检测依赖：列出依赖摘要（做不到也至少提示“可能影响下游”）
2) 批量重命名/删除后，若会影响项目配置引用：提示“已生成引用更新报告/可下载”

## 4) 下一步怎么做（“先调”的交付物）
为了让“先调”可验收，建议你们输出这 3 个交付物（不要求大重构）：
1) **按钮归属表（全量）**：把项目相关页面的每个按钮填入《页面-按钮-对象归属清单》
2) **文案/分组调整 PR**：仅做 UI 分组与文案，不改业务逻辑（或极小改动）
3) **状态面板补齐**：在项目中心明确三状态：配置/数据库/加载

验收标准（用户视角）：
- 用户不会再纠结“保存为项目/重建数据库/加载到SQLite/加载数据库”的区别
- 做完一次“源文件关系调整 + 重导”，不会发生“库变了但关系没保存”
- 用户任何时候都能在项目中心看清：现在是谁（项目）、库有没有、库加载没
